# Doom port for a Renesas Synergy microcontroller platform

This project is a port of [Doom](https://en.wikipedia.org/wiki/Doom_(1993_video_game)) to the Renesas Synergy S7G2 microcontroller. The goal is to run Doom on the board's integrated TFT display, without external SDRAM, with the main game file embedded in QSPI flash, and with a custom backend for video, touch, and diagnostics.

The project was tuned for the board's real profile: limited internal RAM, display controlled by `GLCDC + ILI9341`, startup through `ThreadX`, and direct integration with files generated by the Synergy ecosystem.

## Port goals

- Run Doom on the integrated TFT display of the `SK-S7G2`.
- Reuse the `doomgeneric` engine as the game base.
- Store `doom1.wad` in QSPI flash, avoiding RAM consumption at boot.
- Operate without external SDRAM.
- Provide a simple debug path through the global variable `doom_error_msg`.

## Architecture summary

The project is split into four main layers:

1. **Board and RTOS startup**  
   Files generated by Synergy and SSP, such as `src/synergy_gen`, `synergy_cfg/ssp_cfg`, and `configuration.xml`, configure `ThreadX`, `GLCDC`, `SCI/SPI`, `RIIC`, `DTC`, pins, and BSP heap.

2. **Renesas backend for the port**  
   File `src/doomgeneric_renesas.c` implements the interface expected by `doomgeneric`:
   - `DG_Init()`
   - `DG_DrawFrame()`
   - `DG_GetKey()`
   - `DG_SleepMs()`
   - `DG_GetTicksMs()`
   - `DG_UpdatePalette()`

3. **LCD initialization and control**  
   Files `src/lcd_setup.c` and `src/lcd.h` configure the `ILI9341` over SPI and prepare the panel to receive the framebuffer produced by `GLCDC`.

4. **Doom engine**  
   Directory `src/doomgeneric` contains the main Doom codebase. The game entry point is the main thread in `src/main_thread_entry.c`.

## Execution flow

Current boot flow:

1. The project starts `ThreadX` and initializes instances generated by Synergy.
2. `main_thread_entry()` clears `doom_error_msg` and calls `doomgeneric_Create()`.
3. `DG_Init()` initializes display and touch.
4. `doomgeneric_Create()` boots Doom with a reduced command line.
5. The infinite loop calls `doomgeneric_Tick()`.
6. Each frame goes through the Doom renderer and reaches the `GLCDC` framebuffer.

## Current game command line

The game boots directly with a simplified profile, defined in `src/main_thread_entry.c`:

- `-iwad doom1.wad`
- `-skill 1`
- `-warp 1 1`
- `-nomonsters`
- `-nosound`
- `-nomusic`
- `-nogui`

This profile was chosen to reduce CPU load, RAM usage, and external dependencies during port validation.

  BSP and SSP module configuration.

## Current status

In the repository's current state:

- the game already boots and renders on the integrated display;
- `doom1.wad` is loaded from QSPI flash;
- `S4` and `S5` work as fire buttons;
- touch input already responds, but still needs geometry and calibration fine tuning to be ideal in all directions;
- `doom_error_msg` is the main observation point when something fails during boot or runtime.

## Quick diagnostics

If the board hangs, shows a black screen, or stops responding:

1. pause execution in the debugger;
2. inspect `doom_error_msg`;
3. confirm which stage the boot stopped at;
4. follow the corresponding themed documentation.

Most useful error messages usually appear as:

- boot stage markers, for example `stage: ...`;
- memory allocation failures;
- touch/I2C failures;
- `HardFault`, `BusFault`, `MemManage`, or `UsageFault`.

## Important limitations and notes

- The board **does not have external SDRAM**.
- The port depends heavily on balance between framebuffer, BSP heap, and Doom zone.
- Part of the project lives in Synergy-generated files; careless regeneration may overwrite important display, touch, and peripheral adjustments.
- The board touch is handled here as a low-level input path, without using the full GUIX stack.
