# Build, Flash, and Debug

This document summarizes the recommended flow to build, flash, and debug the project in `e2 studio`.

## Prerequisites

- `Renesas e2 studio` with Synergy project support;
- `GNU Arm Embedded` toolchain compatible with the project;
- `SK-S7G2` board;
- debugger configured in the e2 studio environment.

## Files that most affect the build

- `configuration.xml`  
  Main Synergy project configuration.
- `src/synergy_gen`  
  Code generated by the configurator.
- `synergy_cfg/ssp_cfg`  
  BSP configuration, heap, stacks, and module parameters.
- `src`  
  Port and engine source code.
- `script/r7fs7g27h3a01cfc.ld`  
  Linker script.

## Recommended flow in e2 studio

1. open project `DoomRenesas`;
2. confirm `configuration.xml` is coherent with generated code;
3. avoid regenerating Synergy files without reviewing display, touch, `IIC2`, SPI, and pins afterward;
4. select configuration `Debug`;
5. build the project;
6. flash the application to the board;
7. open a debug session if needed.

## Generated artifacts

In configuration `Debug`, the build typically generates:

- `Debug/DoomRenesas.elf`
- `Debug/DoomRenesas.srec`
- `Debug/DoomRenesas.map`

The `.elf` is the main debugging artifact. The `.srec` is useful for flashing. The `.map` is essential when memory usage is in question.

## Command-line build

The project also exposes a `make` infrastructure in the `Debug` folder. In an environment configured by e2 studio, the build usually follows:

```powershell
make -r -j16 all
```

Even so, for this project, the safest flow remains e2 studio itself, because integration with Synergy-generated files must stay consistent.

## Embedded WAD

The project expects `doom1.wad` already converted to a C array in:

- `src/doom1_wad.c`

This file is treated as an embedded asset. Therefore:

- there is no runtime file system dependency for this WAD;
- changes to WAD contents require a firmware rebuild.

## Recommended debug flow

When the game does not boot correctly:

1. start debug;
2. halt CPU;
3. inspect `doom_error_msg`;
4. use this message to identify whether failure was:
   - boot;
   - display;
   - touch;
   - allocation;
   - CPU fault.

## Useful breakpoint points

- `main_thread_entry()`
- `doomgeneric_Create()`
- `DG_Init()`
- `DG_UpdatePalette()`
- `DG_DrawFrame()`
- `doom_set_error_literal()`
- `doom_set_error_msg()`

## How to interpret `doom_error_msg`

`doom_error_msg` is the center of the project's debugging strategy.

It may record:

- stage names, such as `stage: ...`;
- Doom zone allocation failures;
- LCD, SPI, or touch errors;
- CPU faults such as `HardFault` and `BusFault`.

Since the variable stays in `.noinit`, it also helps in fault scenarios where the program cannot produce more elaborate logging.

## Care with generated files

An important project note:

- several critical components depend on `src/synergy_gen` and `synergy_cfg/ssp_cfg`;
- regeneration may overwrite callbacks, vectors, and fine-tuned settings;
- after regeneration, review at least:
  - `main_thread.c/.h`
  - `common_data.c/.h`
  - `pin_data.c`
  - `bsp_cfg.h`

## When to inspect the `.map` file

Open `Debug/DoomRenesas.map` when you need to confirm:

- `.bss` usage;
- heap usage;
- framebuffer size;
- embedded WAD placement;
- unexpected memory growth after a change.
